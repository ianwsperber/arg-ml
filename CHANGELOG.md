# Changelog

All notable changes to the ArgML reference implementation are recorded here. Each entry corresponds to a phase from [`PLAN.md`](./PLAN.md).

## v0.2 Renderer Redesign — Phases 0–7

The single-file HTML renderer was redesigned to surface ArgML 0.2 features (reader overlays, takeaways, attitudes, propagation, same-as classes, modes, generators) and to let a reader edit attitudes live with the canonical propagation engine in the loop. The pill-style sticky toolbar from prior phases is preserved; the v2 design drafts in `docs/project/v2-designs/` informed the layout (three-column reader grid, in-margin attitude controls, bottom reader drawer) but their client-side `statusOf` was rejected in favour of reusing `src/propagation/`.

- **Phase 0 — Build tooling.** Replaced the regex-strip bundler in `scripts/generate-render-assets.ts` with an esbuild IIFE (browser target) so `arg-render.ts` can import `src/propagation/`, `src/parser/`, and `src/ast/` directly. New `arg-render.bootstrap.ts` entry calls `mount()`. `tsconfig.client.json` is a type-check gate ensuring the client bundle stays browser-safe (no Node imports).
- **Phase 1 — Overlay plumbing.** `RenderOptions.overlaySource?: string` added to `renderHTML`. When supplied, the overlay is base64-embedded as `<script id="argml-overlay" type="application/argml-overlay-b64">`. `decodeArgmlPayload` decodes both post and overlay payloads at mount. `<html data-has-overlay="true|false">` lets CSS branch on overlay presence.
- **Phase 2 — Canonical propagation.** Server-side `propagate()` is run during `renderHTML` and its result is embedded as `<script id="argml-initial-status" type="application/json">` (postPrefix + takeaway statuses + node statuses) so the first paint shows correct statuses *before* JS rehydrates. Client `mount()` parses the post + overlay AST in the browser and re-runs `propagate()` on every attitude edit.
- **Phase 3 — Takeaways strip + status-tinted prose.** `<section class="takeaways-strip">` below the masthead with one `.takeaway-row` per head-level `<takeaway>` showing priority, ref, summary, and a status pill. `applyTakeawayStatuses` hydrates rows with a "blocked by … / open via …" cause line linking to the rejected/open ancestors. Status-tinted underlines on `.ann-claim[data-status=...]` via four CSS color tokens (endorsed / supported / provisional / blocked).
- **Phase 4 — Per-atom attitude controls.** Each gloss row carries `<div class="gloss-actions">` with three buttons (✓ accept / ✕ reject / ? open) emitted by `buildAttitudeButtons`. `setAttitude` mutates an in-memory `attitudes: Map<localId, AttitudeKind>`, calls `applyAttitudeToDom`, and triggers a `recompute()` that synthesises an overlay AST, runs `propagate()`, and stamps the new statuses back onto atoms + takeaway rows. Clicking the active kind clears the attitude.
- **Phase 5 — Left-gutter graph.** Always-on pale bezier edges drawn into `.graph-svg` keyed off `[data-node-id]` (separate namespace from `[data-id]` on prose). Edges typed by relation (supports / attacks / rests-on / via / same-as). The `Graph` toolbar pill gates the full label panel; edges themselves are visible at all times, lighting up in typed colors when the related node is hovered. Same-as restatements drawn as dashed blue edges (hover-only to limit visual noise).
- **Phase 5b — Feedback.** Fixes from a live review pass: programmatic `.mode-tooltip` (native `title` was intercepted by the gloss popup); whole `.takeaway-row` is clickable and scrolls to its claim; argument / inference blocks flow as paragraphs with a left rule rather than block-quote styling.
- **Phase 6 — Reader drawer + Reader toolbar pill.** Fourth `data-toggle="reader"` pill; `setupToolbar` initialises `data-mode-reader="off"` so the drawer starts hidden. `renderReaderDrawer()` emits `.reader-drawer` with a two-column grid: **Marks** (`<li.drawer-mark>` per active attitude, with kind label, claim summary, and × clear button) and **Implications** (`<li.drawer-impl>` per takeaway whose status was shifted by the reader's marks, showing the engine's `rejectedAncestors` / `openAncestors` as `#claim-*` links). The **Export overlay XML** button calls `serializeReaderOverlay` from `src/parser/serialize.ts` (verified pre-existing) and triggers a download as `<postId>.overlay.xml` (Blob URL with a data: URL fallback). Drawer rebuilds on every attitude edit via `rebuildReaderDrawer(root, attitudes, lastResult, state.claims)`.
- **Phase 7 — Polish (mobile, a11y, perf).** New exported `applyStatusDiff(root, prev, next)` lets `applyPropagationResult` accept an optional `prev` map and only mutate atoms whose status changed — each click goes from O(n) to O(changes) atom writes. `mount()` caches `lastStatusMap` (seeded from `initialStatus`) and feeds it back into each recompute. Claim atoms now render with `tabindex="0"`, `role="button"`, and an `aria-label` describing the shortcuts; a global keydown handler maps `1` → accept, `2` → reject, `3` → open on the focused `.ann-claim` (toggle clears). Modifier keys (ctrl/meta/alt) suppress the hotkeys. `.ann-claim:focus-visible` adds a clear focus outline. Toolbar at ≤520px drops the `ArgML` brand label and tightens pill padding so all four pills fit phone widths.
- **Examples.** `examples/morality-without-consciousness.overlay.xml` (Appendix B.2 verbatim, from Phase 4.3) drives a second rendered output `examples/rendered/morality-without-consciousness.with-overlay.html`. `scripts/render-examples.ts` discovers sibling `.overlay.xml` files and passes them as `overlaySource`.
- **Tests.** 23 new tests across the renderer redesign land in `src/render/assets/arg-render.test.ts` (now 108 tests in this file, ~295 total). Coverage spans every helper (`applyStatusDiff`, `buildAttitudeButtons`, `buildAttitudeCard`, `buildImplicationItem`, `synthesizeOverlay`, `collectOverlayAttitudes`, `applyAttitudeToDom`, `diffStatuses`, `rebuildReaderDrawer`, `serializeAttitudesAsOverlay`) plus happy-dom integration through `mount()` for: pill toggles, reject-click-blocks-takeaway, drawer mark + implication wiring, mark-clear button, overlay-export download (with a `URL.createObjectURL` stub to capture the Blob), keyboard-shortcut toggles, modifier-key suppression, focusability, and diff-skip-unchanged behaviour. Server-side, `src/render/html.test.ts` covers the overlay payload embedding and the initial-status JSON.
- **Out of scope.** Cross-document propagation across fetched imports (still Phase 7 of the *original* PLAN); LLM-assisted Markdown-to-ArgML (Phase 5 of the original PLAN, open in PR #12); an architecture ADR for this redesign (deferred — load-bearing decisions are summarised in the redesign plan).

## Phase 4.4 — Local Propagation Engine

- **Propagation module** (`src/propagation/`, new). Three internal files plus a barrel:
  - `equivalence.ts` — Union-find over local `same-as` edges produces equivalence classes of claim ids. Cross-document `same-as` (`prefix:id`) references are not merged (Phase 7 territory). The returned interface exposes `classOf`, `members`, `equivalent`.
  - `graph.ts` — Builds the propagation graph (spec §13.5). Edges are premise → consequent: `claim.supports`, `argument.supports`, `inference.from`/`.to`, `rests-on`. The `via` attribute on a `<claim>` is treated as a reference annotation rather than a graph edge — `inference.to` is the authoritative consequent edge. Ancestor sets are merged across same-as classes so a single attitude propagates to every co-referenced node.
  - `propagate.ts` — Implements the §13.5 four-status classification: `endorsed` / `supported` / `provisional` / `blocked`. Mode-aware: rejection of an `anticipated-objection`, `attributed`, `reductio-target`, or `conceded` claim is non-blocking — keyed off the *targeted* claim's mode rather than whichever same-as class member the BFS reaches (see `SPEC-NOTES.md` for the worked example). Rejection of an `<inference>` or `<assumption>` always blocks. Rejection of an `<argument>` only blocks when the argument directly supports the takeaway AND no alternate path to a leaf remains (`hasAlternatePath` BFS with the rejected node excluded). The overlay's import prefix mapping to the post is auto-detected by matching `post.id` against a `/`-delimited path segment of each import's `doc` URL; ambiguous matches emit `PROP002`, no match emits `PROP001`, and an unresolved attitude target emits `PROP003`. An explicit `postPrefix` option overrides auto-detection.
- **CLI** (`src/cli/`). New `argml propagate <post> --overlay <overlay> [--format text|json] [--prefix <p>]` subcommand. Text output renders a takeaways table with columns ID / PRIORITY / STATUS / REJECTED / OPEN; JSON output emits a machine-readable payload (post id, reader, prefix, diagnostics, takeaways with full ancestor lists). `argml validate <file> --overlay <overlay>` validates both documents and additionally reports overlay targets that fail to resolve in the post (emitting cross-document diagnostics as a third block).
- **Worked example fix.** Added `rests-on="C4.5"` to `<claim id="C4.9">` in `examples/morality-without-consciousness.argml.xml`. The spec's Appendix B propagation table treats `C4.5` as an ancestor of `C4.9`, but the example as previously written had no graph edge between them — the dependency was only readable from the prose. The new `rests-on` edge makes the dependency explicit, and the integration test now produces the spec's expected results verbatim: `C6.7: provisional`, `C4.9: provisional`, `C3.6: blocked`.
- **Public API additions** (`src/index.ts`): `propagate`, `computeEquivalenceClasses`, `buildPropagationGraph`, and the supporting types (`PropagationResult`, `NodeStatus`, `TakeawayStatus`, `PropagationStatus`, `PropagationDiagnostic`, `PropagationDiagnosticCode`, `PropagationOptions`, `PropagationGraph`, `PropagationNode`, `PropagationNodeKind`, `EquivalenceClasses`).
- **Tests.** 33 new tests across `src/propagation/equivalence.test.ts` (5), `src/propagation/graph.test.ts` (7), `src/propagation/propagate.test.ts` (16 — covering each status, every mode-shielding rule, same-as propagation in both directions (target-mode wins over visited-node mode and vice-versa), both PROP-diagnostic codes, the canonical Appendix B integration assertion, and two monotonicity property tests), and `src/cli/propagate.test.ts` (3). Suite is now 235 tests, all green. `pnpm typecheck && pnpm test && pnpm lint` clean.
- **Diagnostic codes.** Three propagation diagnostics (`PROP001`–`PROP003`) live alongside the four-status output rather than in the validator's `ARGML…` / `OVERLAY…` tables — they describe the matching between a post and an overlay, not document well-formedness.
- **Out of scope** (per Phase 4.4 deliverables): cross-document propagation across fetched imports (Phase 7); visual overlay rendering atop posts (Phase 6); richer than-minimum Dung-style argumentation semantics (informative aside in spec §14.2; the four-status classification is the minimum behavior).

## Phase 4.3 — Reader-Overlay Document Type

- **AST** (`src/ast/overlay.ts`, new). New top-level discriminated union `ParsedDocument = ArgMLDocument | ReaderOverlayDocument` in `src/ast/index.ts`. `ReaderOverlayDocument` carries `reader`, optional `updated`, a required `ImportsNode`, and arrays of `AttitudeNode` and `SubstitutionNode`. `AttitudeNode` records `target` (`prefix:id`), `attitudeKind` (`accept` | `reject` | `open`), optional `rejectionType`, optional `credence`, and a `note: InlineNode[]` for the reader's commentary. `SubstitutionNode` mirrors the shape with `target` and `use`. Closed-enum alias `AttitudeKind` and runtime const `ATTITUDE_KINDS` provided.
- **Parser** (`src/parser/overlay-builder.ts`, new; `src/parser/parse.ts`). New root-dispatching `parse(xml)` returns either a post or an overlay. `parseReaderOverlay(xml)` typed entry point for callers that want overlays specifically; `parseArgML(xml)` now errors with `PARSE003` when handed an overlay root. Builder is structurally parallel to `builder.ts`: namespace check, attribute extraction, per-section iteration. Note content is parsed via the shared `buildInlineChildren` helper (now exported) so any presentational/inline markup in the reader's prose flattens consistently. New parse diagnostics: `PARSE014` (missing `reader`), `PARSE015` (`<attitude>` missing `target`/`kind` or unknown kind), `PARSE016` (`<substitution>` missing `target`/`use`).
- **Serializer** (`src/parser/serialize.ts`). `serializeReaderOverlay(doc)` round-trips the overlay shape, preserving every attribute and inline-text note content. The Appendix B.2 example round-trips: parse → serialize → re-parse yields a structurally equivalent overlay.
- **Validator** (`src/validator/overlay.ts`, new; `src/validator/validate.ts`, `codes.ts`). New `validateOverlay(overlay)` function plus a dispatching `validateAny(ParsedDocument)` in `validate.ts`. Added codes `OVERLAY001`–`OVERLAY008`: duplicate attitude target (error), reject without `rejection-type` (error), non-reject with `rejection-type` (warn), undeclared prefix on attitude target (error), no `prefix:` segment on overlay target (warn), undeclared prefix on substitution target/use (error), duplicate substitution target (warn), numeric credence outside [0, 1] or with spurious precision (warn). Cross-document target resolution against actual post ids is deferred to Phase 7.
- **CLI** (`src/cli/`). `argml validate <file>` now accepts both document types via a new dispatching `loadAnyDocument` and `validateAny`. `argml summary <file>` dispatches: posts print the existing summary; overlays print reader id, updated date, attitude counts by kind, substitution count, imports, and a sorted list of `(prefix, target)` pairs. New `argml overlay show <file>` subcommand pretty-prints attitudes and substitutions in two tabular views. The strict `loadDocument` (post-only) remains for callers that need it.
- **Example.** `examples/morality-without-consciousness.overlay.xml` is the spec Appendix B.2 reader-overlay verbatim — 6 attitudes (accept, reject/undercut, reject/rebut, open) and 1 substitution against the upgraded post. Parses and validates clean (0 errors, 0 warnings).
- **Tests.** 29 new tests across `src/parser/overlay.test.ts` (parser, dispatcher, round-trip, Appendix B.2 integration), `src/validator/overlay.test.ts` (one positive case per OVERLAY code), and `src/cli/overlay.test.ts` (validate, summary, `overlay show`, refusal of post root). Suite is now 202 tests, all green. `pnpm typecheck && pnpm test && pnpm lint` clean.
- **Public API additions** (`src/index.ts`): `parse`, `parseReaderOverlay`, `serializeReaderOverlay`, `validateOverlay`, `validateAny`, types `AnyParseResult`, `OverlayParseResult`, `ReaderOverlayDocument`, `AttitudeNode`, `SubstitutionNode`, `AttitudeKind`, `ATTITUDE_KINDS`, `ParsedDocument`.
- **Out of scope** (per Phase 4.3 deliverables): propagation analysis (lands in 4.4), visual rendering of overlays alongside posts (Phase 6), cross-document target resolution against fetched imports (Phase 7).

- **Tests.** 32 new tests added across `src/validator/validate.test.ts` (one positive + one negative per new code where the negative case is non-trivial) and the new `src/parser/parse02.test.ts` (head additions, body additions, round-trip parity). - **AST shape change (note for downstream consumers).** `ClaimNode.provenance`, `InferenceNode.provenance`, `ConflictNode.provenance`, `TermDeclaration.provenance`, and `AssumptionNode.provenance` are declared as required `string[]` (empty when no provenance is recorded) rather than optional. ## Phase 4 — HTML Renderer

- Added `renderHTML(doc, options): string` in `src/render/html.ts`. Output is a self-contained HTML5 document that embeds (a) the raw ArgML XML inside a `<script type="application/xml" id="argml-source">` payload, (b) the bundled stylesheet inline, and (c) a bundled client renderer script inline. No external assets are fetched. Re-exported from `src/index.ts`.
- The visible reading experience is produced **client-side** by `src/render/assets/arg-render.ts`, which parses the embedded XML via `DOMParser`, walks the tree, and mounts the rendered document into `<div id="root">`. Features:
  - Visual encoding: dotted-underline terms; numbered claim anchors; color-coded credence and strength badges; defeasible inferences as left-bordered block asides (double-border for strict); conflicts as bordered asides with attacker → target pair and optional response; evidence as hoverable superscripts; `<epistemic-status>` banner at the top.
  - Interactive reader: hover tooltips for terms / claims / inferences / evidence / notes; a right-gutter marginalia view with glosses and SVG arrows linking related claims; a left-gutter graph view of the argument structure with hover-to-highlight; a toolbar to toggle frontmatter / annotations / graph modes.
  - Cross-document `<term ref="prefix:id">` references render with an `argml-external` marker pending Phase 7 resolution (logged in `SPEC-NOTES.md`).
- **Architectural divergence from the original PLAN.** Project.md §Phase 4 specified server-side templates with markup that remains readable when CSS is disabled. This phase ships a JS-driven client renderer instead — the body is parsed and rendered in the browser. With JavaScript disabled the page is a blank shell. Recorded in `SPEC-NOTES.md` and `docs/adr/0001-client-side-html-renderer.md`; Project.md §Phase 4 acceptance criteria have been updated to match.
- Client renderer is TypeScript (`src/render/assets/arg-render.ts`) compiled by a dedicated `tsconfig.client.json` to a single browser-targeted script, then baked into `src/render/assets.generated.ts` by `scripts/generate-render-assets.ts`. The generator runs automatically via `pretest` / `prebuild`, so the inlined bundle cannot silently drift from its source.
- Pure rendering helpers (`escape`, `parseList`, `mdLinks`, `refLabel`, `buildClaimGloss`, `buildTermGloss`, `buildInferenceGloss`, `buildEvidenceGloss`, `renderFrontmatter`) live in `src/render/assets/arg-render-core.ts` so they can be unit-tested without a DOM.
- Replaced the Phase 3 `runRender` stub in `src/cli/render.ts`: loads the document, renders, writes to `--output <path>` or stdout; non-zero exit on parse failure or file write error.
- Implemented `pnpm render-examples` (script at `scripts/render-examples.ts`) — iterates `examples/*.argml.xml`, runs parse + validate + renderHTML, writes `examples/rendered/*.html`, exits non-zero on parse failure.
- New dev dependencies: `tsx` (run TypeScript scripts), `happy-dom` (DOM environment for renderer tests).
- Tests: pure-helper unit tests in `src/render/assets/arg-render-core.test.ts`; integration tests in `src/render/assets/arg-render.test.ts` that mount the renderer in `happy-dom` against the worked example and assert document structure (header, prose annotations, gloss gutter, graph nodes, parsererror fallback, external-ref handling). Existing tests in `src/render/{escape,html}.test.ts` and `src/cli/render.test.ts` retained.

## Phase 1 — Core Data Model and Parser

- Added typed AST in `src/ast/` covering every element in spec §7, with discriminated unions for inline body nodes and distinct kinds for declaration-form (`term-decl`) vs reference-form (`term-ref`) `<term>`.
- Implemented `parseArgML(xml)` in `src/parser/` on top of `fast-xml-parser` (`preserveOrder` + `captureMetaData`), returning a `ParseResult` with a typed document and a diagnostic array (never throws on user input).
- Source positions on every AST node, sourced from the parser's `startIndex` metadata and a precomputed `LineMap`. Covers at least `<claim>`, `<term>` (both forms), `<inference>`, and `<conflict>` per Phase 1 acceptance.
- Whitespace handling: preserved inside prose-bearing elements (`<p>`, `<claim>`, body-form `<term>`, `<gloss>`, `<inference>` warrant, etc.); ignored between structural elements.
