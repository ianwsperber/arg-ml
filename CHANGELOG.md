# Changelog

All notable changes to the ArgML reference implementation are recorded here. Each entry corresponds to a phase from [`PLAN.md`](./PLAN.md).

## Phase 4.3 — Reader-Overlay Document Type

- **AST** (`src/ast/overlay.ts`, new). New top-level discriminated union `ParsedDocument = ArgMLDocument | ReaderOverlayDocument` in `src/ast/index.ts`. `ReaderOverlayDocument` carries `reader`, optional `updated`, a required `ImportsNode`, and arrays of `AttitudeNode` and `SubstitutionNode`. `AttitudeNode` records `target` (`prefix:id`), `attitudeKind` (`accept` | `reject` | `open`), optional `rejectionType`, optional `credence`, and a `note: InlineNode[]` for the reader's commentary. `SubstitutionNode` mirrors the shape with `target` and `use`. Closed-enum alias `AttitudeKind` and runtime const `ATTITUDE_KINDS` provided.
- **Parser** (`src/parser/overlay-builder.ts`, new; `src/parser/parse.ts`). New root-dispatching `parse(xml)` returns either a post or an overlay. `parseReaderOverlay(xml)` typed entry point for callers that want overlays specifically; `parseArgML(xml)` now errors with `PARSE003` when handed an overlay root. Builder is structurally parallel to `builder.ts`: namespace check, attribute extraction, per-section iteration. Note content is parsed via the shared `buildInlineChildren` helper (now exported) so any presentational/inline markup in the reader's prose flattens consistently. New parse diagnostics: `PARSE014` (missing `reader`), `PARSE015` (`<attitude>` missing `target`/`kind` or unknown kind), `PARSE016` (`<substitution>` missing `target`/`use`).
- **Serializer** (`src/parser/serialize.ts`). `serializeReaderOverlay(doc)` round-trips the overlay shape, preserving every attribute and inline-text note content. The Appendix B.2 example round-trips: parse → serialize → re-parse yields a structurally equivalent overlay.
- **Validator** (`src/validator/overlay.ts`, new; `src/validator/validate.ts`, `codes.ts`). New `validateOverlay(overlay)` function plus a dispatching `validateAny(ParsedDocument)` in `validate.ts`. Added codes `OVERLAY001`–`OVERLAY008`: duplicate attitude target (error), reject without `rejection-type` (error), non-reject with `rejection-type` (warn), undeclared prefix on attitude target (error), no `prefix:` segment on overlay target (warn), undeclared prefix on substitution target/use (error), duplicate substitution target (warn), numeric credence outside [0, 1] or with spurious precision (warn). Cross-document target resolution against actual post ids is deferred to Phase 7.
- **CLI** (`src/cli/`). `argml validate <file>` now accepts both document types via a new dispatching `loadAnyDocument` and `validateAny`. `argml summary <file>` dispatches: posts print the existing summary; overlays print reader id, updated date, attitude counts by kind, substitution count, imports, and a sorted list of `(prefix, target)` pairs. New `argml overlay show <file>` subcommand pretty-prints attitudes and substitutions in two tabular views. The strict `loadDocument` (post-only) remains for callers that need it.
- **Example.** `examples/morality-without-consciousness.overlay.xml` is the spec Appendix B.2 reader-overlay verbatim — 6 attitudes (accept, reject/undercut, reject/rebut, open) and 1 substitution against the upgraded post. Parses and validates clean (0 errors, 0 warnings).
- **Tests.** 29 new tests across `src/parser/overlay.test.ts` (parser, dispatcher, round-trip, Appendix B.2 integration), `src/validator/overlay.test.ts` (one positive case per OVERLAY code), and `src/cli/overlay.test.ts` (validate, summary, `overlay show`, refusal of post root). Suite is now 202 tests, all green. `pnpm typecheck && pnpm test && pnpm lint` clean.
- **Public API additions** (`src/index.ts`): `parse`, `parseReaderOverlay`, `serializeReaderOverlay`, `validateOverlay`, `validateAny`, types `AnyParseResult`, `OverlayParseResult`, `ReaderOverlayDocument`, `AttitudeNode`, `SubstitutionNode`, `AttitudeKind`, `ATTITUDE_KINDS`, `ParsedDocument`.
- **Out of scope** (per Phase 4.3 deliverables): propagation analysis (lands in 4.4), visual rendering of overlays alongside posts (Phase 6), cross-document target resolution against fetched imports (Phase 7).

## Phase 4.2 — Post-Document 0.2 Extensions

- **AST.** `src/ast/nodes.ts` extended with the closed-enum aliases `ClaimMode`, `ArgumentMode`, `InferencePattern` (each accompanied by a runtime `*_MODES`/`*_PATTERNS` const used by validators and tooling). New optional fields on `ClaimNode`: `mode`, `attributedTo`, `sameAs`, `source`, `provenance`. New optional fields on `InferenceNode`: `pattern`, `provenance`. Added `provenance: string[]` to `ConflictNode`, `TermDeclaration`, `AssumptionNode`. New types `GeneratorNode`, `ProvenanceNode`, `TakeawayNode`, `TakeawaysNode`, `ArgumentNode`. `HeadNode` extended with optional `provenance` and `takeaways`. `BlockOrInline` extended to include `ArgumentNode`. All new fields are optional or default-initialised so 0.1 documents continue to parse cleanly.
- **Parser** (`src/parser/builder.ts`). New builders for `<provenance>`/`<generator>`, `<takeaways>`/`<takeaway>`, and `<argument>`. `buildHead` now enforces the spec-ordered sequence (metadata → provenance → imports → terms → assumptions → takeaways) via a new `HEAD_CHILD_ORDER` map. `buildClaim` parses `mode`, `attributed-to`, `same-as`, `source`, `provenance`. `buildInference` parses `pattern`, `provenance`. `buildAssumption`, `buildConflict`, `buildTermDecl` parse `provenance`. New parse diagnostics: `PARSE010` (head children out of order), `PARSE011` (`<argument>` missing `mode`), `PARSE012` (`<takeaway>` missing `ref`), `PARSE013` (`<generator>` missing `id`).
- **Serializer** (`src/parser/serialize.ts`). Round-trips every new attribute and element (mode, same-as, attributed-to, source, pattern, provenance, generators, takeaways, arguments).
- **Validator** (`src/validator/validate.ts`, `codes.ts`). Added codes `ARGML017`–`ARGML030` covering: unknown claim mode (warn), `mode="restated"` requires `same-as` (error), `mode="reductio-target"` should pair with `defeasible="false"` on the licensing inference (warn), `mode="attributed"` should carry `attributed-to` (warn), unknown inference pattern (warn), takeaway must resolve to a local `<claim>` (error), duplicate takeaway (warn), provenance id must resolve to a `<generator>` (error), `same-as` resolution (warn), `same-as` cycle (warn), argument-supports kind mismatch (warn), inference `from` referencing an `<argument>` outside `argument-by-cases` (warn), unknown argument mode (warn). New `SymbolKind` values `argument` and `generator`. Inference `from` now accepts `<argument>` references (alongside claims and assumptions).
- **CLI.** `argml summary` reports counts for arguments, takeaways, generators; lists non-`asserted` claim-mode distribution and inference-pattern distribution; lists takeaways with priority. `argml graph` (JSON + DOT) emits `<argument>` as graph nodes (parallelogram in DOT), encodes `mode`, `pattern`, `argumentMode` on nodes, and adds the `via-argument` and `same-as` edge kinds (the latter rendered as a blue dashed edge in DOT). `argml deps` walks `<argument>` supports/rests-on alongside claim relations. `walk.ts` collects `arguments`, `takeaways`, `generators`.
- **Renderer** (`src/render/assets/arg-render.ts`). `ClaimRec` and `InferenceRec` extended with the new attributes; new `ArgumentRec`, `GeneratorRec`, `TakeawayRec` types; `RenderState` now includes `arguments`, `generators`, `takeaways`. New collectors: `collectGenerators`, `collectTakeaways`. `renderNode` handles `<argument>` (renders as `<aside class="argument-block" data-mode="…">` with a head row showing mode + id + supports list); claim rendering surfaces `data-mode`, `data-same-as`, `data-attributed-to`, `data-provenance`; inference rendering surfaces `data-pattern` and `data-provenance` and includes pattern in the inline meta line. `renderFrontmatter` adds Takeaways and Provenance blocks. `buildClaimGloss` and `buildInferenceGloss` show mode badge, same-as link, attributed-to, source, pattern, and provenance chain. `arg-render.css` extended with `.argument-block`, `.mode-badge`, `.fm-takeaways`, `.fm-provenance`, `.gloss-row .prov`, plus inline mode hint via `.ann-claim[data-mode]::before` and same-as marker via `.ann-claim[data-same-as]::after`.
- **Worked example.** `examples/morality-without-consciousness.argml.xml` regenerated against ArgML 0.2 — exercises every new construct (5 modes on claims, 3 inference patterns, 5 arguments across `case` and `thought-experiment` modes, 3 takeaways with all 3 priorities, 2 generators with `human` / `llm` types). Validates clean (0 errors, 0 warnings); rendered HTML regenerated under `examples/rendered/`.
- **Tests.** 32 new tests added across `src/validator/validate.test.ts` (one positive + one negative per new code where the negative case is non-trivial) and the new `src/parser/parse02.test.ts` (head additions, body additions, round-trip parity). The full suite is now 168 tests, all green; existing 0.1 tests unchanged in behaviour. `pnpm typecheck && pnpm test && pnpm lint` clean.
- **AST shape change (note for downstream consumers).** `ClaimNode.provenance`, `InferenceNode.provenance`, `ConflictNode.provenance`, `TermDeclaration.provenance`, and `AssumptionNode.provenance` are declared as required `string[]` (empty when no provenance is recorded) rather than optional. Code that constructs these nodes directly must initialise the field — the parser does this automatically. Same goes for `ArgumentNode.provenance`, `ArgumentNode.disallowedAttrs`, and `TakeawayNode.provenance`. The required-empty-array shape is intentional: it lets every consumer iterate provenance unconditionally without `?? []` boilerplate.

## Phase 4 — HTML Renderer

- Added `renderHTML(doc, options): string` in `src/render/html.ts`. Output is a self-contained HTML5 document that embeds (a) the raw ArgML XML inside a `<script type="application/xml" id="argml-source">` payload, (b) the bundled stylesheet inline, and (c) a bundled client renderer script inline. No external assets are fetched. Re-exported from `src/index.ts`.
- The visible reading experience is produced **client-side** by `src/render/assets/arg-render.ts`, which parses the embedded XML via `DOMParser`, walks the tree, and mounts the rendered document into `<div id="root">`. Features:
  - Visual encoding: dotted-underline terms; numbered claim anchors; color-coded credence and strength badges; defeasible inferences as left-bordered block asides (double-border for strict); conflicts as bordered asides with attacker → target pair and optional response; evidence as hoverable superscripts; `<epistemic-status>` banner at the top.
  - Interactive reader: hover tooltips for terms / claims / inferences / evidence / notes; a right-gutter marginalia view with glosses and SVG arrows linking related claims; a left-gutter graph view of the argument structure with hover-to-highlight; a toolbar to toggle frontmatter / annotations / graph modes.
  - Cross-document `<term ref="prefix:id">` references render with an `argml-external` marker pending Phase 7 resolution (logged in `SPEC-NOTES.md`).
- **Architectural divergence from the original PLAN.** Project.md §Phase 4 specified server-side templates with markup that remains readable when CSS is disabled. This phase ships a JS-driven client renderer instead — the body is parsed and rendered in the browser. With JavaScript disabled the page is a blank shell. Recorded in `SPEC-NOTES.md` and `docs/adr/0001-client-side-html-renderer.md`; Project.md §Phase 4 acceptance criteria have been updated to match.
- Client renderer is TypeScript (`src/render/assets/arg-render.ts`) compiled by a dedicated `tsconfig.client.json` to a single browser-targeted script, then baked into `src/render/assets.generated.ts` by `scripts/generate-render-assets.ts`. The generator runs automatically via `pretest` / `prebuild`, so the inlined bundle cannot silently drift from its source.
- Pure rendering helpers (`escape`, `parseList`, `mdLinks`, `refLabel`, `buildClaimGloss`, `buildTermGloss`, `buildInferenceGloss`, `buildEvidenceGloss`, `renderFrontmatter`) live in `src/render/assets/arg-render-core.ts` so they can be unit-tested without a DOM.
- Replaced the Phase 3 `runRender` stub in `src/cli/render.ts`: loads the document, renders, writes to `--output <path>` or stdout; non-zero exit on parse failure or file write error.
- Implemented `pnpm render-examples` (script at `scripts/render-examples.ts`) — iterates `examples/*.argml.xml`, runs parse + validate + renderHTML, writes `examples/rendered/*.html`, exits non-zero on parse failure.
- New dev dependencies: `tsx` (run TypeScript scripts), `happy-dom` (DOM environment for renderer tests).
- Tests: pure-helper unit tests in `src/render/assets/arg-render-core.test.ts`; integration tests in `src/render/assets/arg-render.test.ts` that mount the renderer in `happy-dom` against the worked example and assert document structure (header, prose annotations, gloss gutter, graph nodes, parsererror fallback, external-ref handling). Existing tests in `src/render/{escape,html}.test.ts` and `src/cli/render.test.ts` retained.

## Phase 3 — CLI Tool

- Added `argml` CLI in `src/cli/` (built via `tsc` to `dist/cli/main.js`, wired through `package.json` `bin`).
- Subcommands:
  - `argml validate <file>` — runs parser + validator; prints diagnostics as `path:line:col: severity code message`; exits 1 if any errors, 0 otherwise; ends with an `N errors, M warnings` summary line.
  - `argml summary <file>` — structural counts (terms, assumptions, claims, inferences, conflicts, sections, paragraphs), import declarations, and unique cross-document references with their declared-prefix / UNKNOWN PREFIX status.
  - `argml deps <file> --target <id>` — ASCII trees rooted at the target for "rests on", "supports", and "supported by"; cross-doc references render as `[external]`; cycles render as `[cycle]`.
  - `argml graph <file> [--format json|dot]` — cytoscape-shaped JSON (default) or Graphviz DOT. Nodes cover claims, assumptions, inferences, conflicts, and external (cross-doc) references; edges cover supports, attacks (with attack-type), rests-on, via-inference, and inference from/to.
  - `argml render <file> [--output <html>]` — stub for Phase 4; prints `render: not yet implemented (Phase 4)` and exits 0.
- Added `commander@^12.1.0` as a production dependency.
- Unit tests per subcommand plus a clean run against `examples/morality-without-consciousness.argml.xml`.
- CLI modules import only from `src/index.ts` (public API); no deep imports into parser/validator internals.

## Phase 2 — Validation

- Added `validate(doc: ArgMLDocument): Diagnostic[]` in `src/validator/` — a pure, single-pass semantic check over the AST. Never throws; emits diagnostics with stable codes and source positions.
- Introduced diagnostic codes `ARGML001`–`ARGML016` covering: duplicate ids, unresolved local references, undeclared import prefixes, missing inference premises, numeric `credence` / `strength` range and spurious-precision (spec §12.2), empty term aliases, `rests-on` / inference `from` kind mismatch, inference `to` kind, conflict `<attacker>` / `<target>` kind, `strength="deductive"` / `defeasible` coherence, undercut against non-defeasible inferences, term-ref resolution, `via` resolution, and `supports` / `attacks` resolution. Documented in `SPEC-NOTES.md`.
- Cross-document references (`prefix:id`) are checked structurally for prefix declaration only; remote resolution is deferred to Phase 7.
- Diagnostics are returned sorted by `(line, column, code)` for stable test output.
- Added `raw: string` to the numeric variant of `BucketOrNumericValue` (preserves the original lexical form so the validator can detect spurious precision and the serializer can round-trip numeric attributes losslessly).
- Public API: `validate`, `Diagnostic`, `DiagnosticCode`, `ARGML_CODES` re-exported from `src/index.ts`.
- Unit tests cover each `ARGML…` code, the clean-validation of the worked example, and stable diagnostic ordering.

## Phase 1 — Core Data Model and Parser

- Added typed AST in `src/ast/` covering every element in spec §7, with discriminated unions for inline body nodes and distinct kinds for declaration-form (`term-decl`) vs reference-form (`term-ref`) `<term>`.
- Implemented `parseArgML(xml)` in `src/parser/` on top of `fast-xml-parser` (`preserveOrder` + `captureMetaData`), returning a `ParseResult` with a typed document and a diagnostic array (never throws on user input).
- Source positions on every AST node, sourced from the parser's `startIndex` metadata and a precomputed `LineMap`. Covers at least `<claim>`, `<term>` (both forms), `<inference>`, and `<conflict>` per Phase 1 acceptance.
- Whitespace handling: preserved inside prose-bearing elements (`<p>`, `<claim>`, body-form `<term>`, `<gloss>`, `<inference>` warrant, etc.); ignored between structural elements.
- Initial diagnostic codes: `PARSE001` (malformed XML), `PARSE002` (wrong namespace on `<post>`), `PARSE003` (root not `<post>`), `PARSE004` (missing `<head>` or `<body>`), `PARSE005` (unknown element warning). Logged in `SPEC-NOTES.md`.
- Lightweight serializer in `src/parser/serialize.ts` for round-trip testing.
- Added `examples/morality-without-consciousness.argml.xml` (Appendix B worked example).
- Added unit tests covering parse, source positions, and parse → serialize → parse round-trip.
- Public API surface in `src/index.ts` (`parseArgML`, `serializeArgML`, AST types, diagnostic types).
